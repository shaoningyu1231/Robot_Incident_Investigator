# 演示数据脱敏与合成策略

本项目的公开演示数据应为**全合成数据**，只参考真实机器人事故的通用故障模式，不复制、发布或重新包装任何真实 rosbag。目标是在保留技术可信度和跨模态因果关系的同时，确保演示资产不包含公司、客户、人员、地点、设备或内部系统的敏感信息。

对外统一描述：

> This demo uses a fully synthetic robot incident inspired by common real-world failure patterns. It contains no production, customer, employee, or facility data.

## 基本原则

- 真实 rosbag 只用于理解通用故障模式，不进入项目仓库、云端、Gemini 请求或演示设备。
- 不从真实 bag 逐帧复制后再做表面修改；演示数据应从空白场景重新生成。
- 只保留抽象因果关系，例如“障碍物进入安全区 → 安全控制器触发 → 机器人减速停车”。
- 所有公开内容使用虚构名称、虚构坐标、虚构错误码和非生产配置。
- Gemini 接收到的所有输入也必须经过相同检查，不能因“仅发送给 API”而降低标准。
- 无法确认是否敏感的数据默认不使用。

## 可以参考的内容

- ROS topic 和 message type 的通用结构。
- 相机、LiDAR、里程计、速度和日志之间的时间关系。
- 常见机器人故障类型和合理的物理变化趋势。
- 公开标准、开源软件中已有的通用字段与概念。
- 经过重新设计的合理数值范围，但不能复现真实设备的精确参数组合。

## 禁止进入演示数据的内容

- 真实相机帧、录音、人员影像、工牌、屏幕、标签或可识别物品。
- 客户名称、公司内部名称、项目代号、仓库布局和现场照片。
- 真实地图轮廓、导航路线、绝对或局部坐标序列。
- 真实机器人序列号、MAC 地址、IP 地址、主机名和设备 ID。
- 真实 ROS namespace、节点名、frame ID、topic 命名约定和启动参数。
- 内部错误码、日志原文、软件版本、容器名称和文件路径。
- 精确标定参数、传感器外参、控制器参数和安全阈值。
- 员工姓名、账号、邮箱、聊天记录或工单信息。
- 能通过时间、位置、轨迹或多项参数组合反推出真实现场的信息。

## 推荐的合成事故

演示采用一个虚构仓库中的安全停车事件：

- 虚构机器人 `demo_bot_01` 以 `0.8 m/s` 行驶。
- 一个合成纸箱进入机器人前方安全区域。
- 相机图像显示纸箱，合成 LiDAR 测得最近距离 `0.74 m`。
- 演示安全阈值设为 `1.2 m`，仅用于本项目，不代表任何真实设备参数。
- 控制器产生虚构事件 `DEMO_SAFETY_STOP_07`。
- 机器人随后减速到零，形成明确的 ground truth。

示例时间线：

```text
t=40.0s  机器人正常行驶，速度 0.8 m/s
t=41.5s  合成纸箱进入相机视野
t=42.1s  合成 LiDAR 距离低于演示安全阈值
t=42.2s  产生虚构安全停车事件
t=42.8s  机器人速度降到 0
```

## 各类数据的生成策略

### 相机

- 使用 Gazebo、Isaac Sim、Blender、程序生成图像或在可控环境中专门拍摄的无敏感素材。
- 场景、机器人外观、标志和颜色不得复刻真实客户现场。
- 若自行拍摄，确保没有人脸、车牌、地址、屏幕、文件或位置线索。

### LiDAR

- 通过仿真或程序直接生成点云或 LaserScan。
- 使用虚构地图和障碍物几何形状。
- 不从真实扫描中裁剪、旋转或加噪后直接复用。

### 轨迹、里程计和 TF

- 从零生成简化轨迹，例如直线行驶后减速停车。
- 使用虚构坐标系，例如 `demo_map`、`demo_odom`、`demo_base_link`。
- 不保留真实地图原点、路径形状、坐标偏移或标定变换。

### 日志

- 日志全部重新编写，使用虚构节点和错误码。
- 推荐结构化保存为 `logs.jsonl`：

```json
{"timestamp":42.2,"level":"WARN","node":"demo_safety_controller","code":"DEMO_SAFETY_STOP_07","message":"Synthetic obstacle entered the configured demo safety zone."}
```

- 文本中不得包含真实路径、用户名、版本、网络地址或内部术语。

### rosbag 元数据

- 使用新的 bag 名称、时间戳和随机 UUID。
- 清除原始录制主机、真实起止时间、用户路径和构建环境信息。
- topic、frame 和 namespace 使用项目专用的 `demo_` 前缀。

## 建议的数据生成流程

- 从真实经验中只记录一句抽象故障定义，不复制原始数据。
- 写出预期 ground truth 和完整时间线。
- 在独立环境中生成相机、LiDAR、轨迹、速度和日志。
- 将各模态按合成时间戳写入新的 rosbag。
- 从新 bag 导出项目需要的固定格式演示资产。
- 对 rosbag、导出文件、Git 历史和 Gemini 请求内容执行最终检查。
- 通过检查后，只发布合成 bag 和由它生成的资产。

## 发布前检查清单

- [ ] 仓库和 Git 历史中不存在任何真实 rosbag 或真实数据片段。
- [ ] 所有图像均为合成素材或明确为本演示专门制作的安全素材。
- [ ] 地图、轨迹、坐标、TF 和传感器参数均为虚构数据。
- [ ] topic、节点、frame、namespace 和错误码均为演示专用命名。
- [ ] 日志中不存在真实用户名、路径、IP、主机名、版本和内部术语。
- [ ] 文件名、EXIF、bag metadata 和时间戳不包含来源信息。
- [ ] Gemini 请求 payload 中只包含已经批准的合成数据。
- [ ] 项目说明明确标注数据为 fully synthetic。
- [ ] 至少由另一位人员独立检查一次演示资产。
- [ ] 如曾接触公司数据，已确认生成和展示方式符合公司政策。

## 发现问题时的处理

如果发现演示资产可能包含真实信息，应立即停止上传和展示，删除相关构建产物并重新生成。若敏感内容已经进入 Git 历史、云端或外部 API，仅删除当前文件不够，还应按照所属组织的安全流程处理历史清理、访问撤销和事件报告。
